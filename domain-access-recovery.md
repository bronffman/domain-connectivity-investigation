# Domain Access Recovery

## Context

Инцидент произошёл во внутреннем корпоративном сегменте. Рабочая станция находилась в изолированном сетевом контуре организации, где доступ к корпоративным ресурсам, контроллерам домена и внутренним сервисам обеспечивается через статически настроенные сетевые параметры.

Проблема проявлялась в отсутствии доступа к доменной инфраструктуре при сохранении физического сетевого подключения. Из-за невозможности связаться с доменом часть административных операций требовала использования локальных учётных записей вместо доменных.

## Environment

- Рабочая станция Windows 10
- Корпоративный домен Active Directory
- Изолированный внутренний сетевой сегмент
- Статическая конфигурация IPv4
- Коммутатор доступа Cisco

## Initial Symptoms

На рабочей станции отображалось сетевое подключение, но доменные службы были недоступны.

Следующие проверки не пройдены:

```cmd
ping corp.local
nslookup corp.local
nltest /dsgetdc:corp.local
```

Example result:

```text
*** Unknown can't find corp.local: No response from server
Getting DC name failed: Status = 1355 ERROR_NO_SUCH_DOMAIN
```

В Windows также отображалось предупреждение о состоянии сети. Поскольку это была изолированная корпоративная сеть, статус «Нет интернета» в Windows не рассматривался как основная проблема.

## Problem Workstation Configuration

Сетевой адаптер имел статическую конфигурацию IPv4:

```text
IPv4 Address: 10.10.40.164
Subnet Mask: 255.255.255.240
Default Gateway: 10.10.40.161
DNS Servers: Not configured
```

Адаптер подключен и шлюз по умолчанию доступен:

```cmd
ping 10.10.40.161
tracert 10.10.40.161
```

Результат: шлюз ответил успешно.

## DNS Check

Проверка конфигурации DNS проводилась с помощью:

```cmd
netsh interface ipv4 show dnsservers
```

Результат:

```text

```
Это объясняло, почему рабочая станция не могла разрешить доменное имя или обнаружить контроллер домена.

Это подтвердило, что рабочая станция не смогла обнаружить инфраструктуру домена.

### Additional Diagnostics

Были выполнены дополнительные проверки DNS и доменной инфраструктуры:

```cmd
ipconfig /flushdns
ipconfig /registerdns
nslookup corp.local
nltest /dsgetdc:corp.local
```
<img width="938" height="1075" alt="image" src="https://github.com/user-attachments/assets/36fd0a99-2d16-41d4-abc1-90536625348d" />

На последнем скриншоте:

```netsh interface ipv4 show dnsservers
Configuration for interface "Ethernet"
Statically Configured DNS Servers: None
```
вообще не настроен или слетел DNS-сервер.

Из-за этого:

- ping corp.local не работает;
- nltest /dsgetdc:corp.local не находит контроллер домена;
- доменные ресурсы не открываются;
- сетевые диски и приложения могут не работать;
- Windows показывает проблемы с сетью.

При этом сама сеть исправна:

- IP есть 
- шлюз доступен 
- маршруты есть 
- Cisco работает

Для определения корректных параметров DNS была выполнена проверка исправно работающей рабочей станции, расположенной в том же сетевом сегменте.
```
Командой ipconfig /all
Найти строку DNS Servers . . . . . : 10.x.x.x
```

## Route and ARP Checks

Таблица маршрутизации была проверена:

```cmd
route print
```

Рабочая станция имела маршрут по умолчанию через настроенный ею шлюз:

```text
0.0.0.0/0 -> 10.10.40.161
10.10.40.160/28 -> On-link
```

Также была проверена защита от повторного обращения (ARP):

```cmd
arp -a
```

Рабочая станция видела адреса своих локальных шлюзов, поэтому локальное соединение работало.

## Switch Port and Cable Test

Чтобы исключить физическую проблему в сети, кабели были поменяны местами между проблемной рабочей станцией и работающей рабочей станцией на коммутаторе Cisco.
<img width="1221" height="528" alt="image" src="https://github.com/user-attachments/assets/a21c8c67-67bc-4ab4-9146-eb54ed311876" />

Result:

- Рабочая станция продолжала функционировать.
- Проблемная рабочая станция продолжала выходить из строя.

После перестановки кабелей, у нерабочей машины сохранились проблемы, а у рабочей осталось все прежнее.
Значит проблема не в порту/VLAN, а в статических настройках самого нерабочего ПК. Нерабочий ПК вручную настроен неправильно или слетел конфиг. Нужно изменить его IPv4 настройки на такие же по схеме, как у рабочих ПК, но с уникальным свободным IP из сети рабочего ПК.

Conclusion:

```text
Кабель и порт коммутатора не являлись первопричиной проблемы.
Проблема заключалась в рабочей станции или конфигурации её локальной сети.
```

## Administrative Rights Check

Для изменения сетевых параметров рабочей станции требовались права локального администратора.

Поскольку компьютер находился в закрытом контуре и не имел доступа к доменной инфраструктуре, при открытии свойств сетевого адаптера Windows не могла проверить доменные админ-учетки и запрашивала локальную административную учетную запись данного ПК.

Для поиска доступных локальных администраторов была выполнена проверка:

```cmd
net localgroup administrators
```

Пример результата:

```text
Administrator
localadmin1
localadmin2
```

## Findings

В ходе расследования было установлено следующее:

- Сетевой адаптер рабочей станции находился в состоянии подключения.
- Шлюз по умолчанию был доступен и отвечал на запросы.
- Кабель и порт коммутатора Cisco были проверены методом перестановки и исключены из числа возможных причин.
- На рабочей станции отсутствовала корректная конфигурация DNS-серверов.
- Разрешение доменного имени не выполнялось.
- Обнаружение контроллера домена завершалось ошибкой ERROR_NO_SUCH_DOMAIN.
- Параметры IPv4 рабочей станции отличались от конфигурации исправно работающих рабочих станций данного сегмента сети.
- Для внесения изменений в сетевую конфигурацию требовался локальный административный доступ.

На основании выполненных проверок было установлено, что проблема связана с некорректной конфигурацией сетевых параметров рабочей станции, а не с физическим подключением или оборудованием сети.

## Conclusion

В ходе диагностики было установлено, что проблема не связана с кабелем, портом коммутатора или физическим подключением рабочей станции.

Основной причиной являлась некорректная статическая конфигурация сетевого адаптера:

- отсутствовали внутренние DNS-серверы;
- параметры IPv4 отличались от конфигурации исправно работающих рабочих станций;
- рабочая станция не могла разрешить доменное имя и обнаружить контроллер домена.

## Resolution

После получения доступа к локальной учетной записи администратора рабочей станции была выполнена корректировка сетевой конфигурации.

Были выполнены следующие действия:

- выполнена ручная настройка IPv4;
- вручную заданы DNS-серверы организации;
- скорректирован IP-адрес рабочей станции в соответствии с используемым сегментом сети.

Используемые DNS-серверы:

```text
Primary DNS:   10.41.206.21
Secondary DNS: 10.41.207.21
```

Путь изменения настроек:

```

```
На скриншоте показан сетевой адаптер рабочей станции, через свойства которого выполнялась корректировка параметров IPv4 и DNS.

После применения настроек были выполнены контрольные проверки:
```cmd
ipconfig /flushdns
nslookup corp.local
nltest /dsgetdc:corp.local
```

В результате рабочая станция успешно восстановила доступ к домену и корпоративным ресурсам.
